### 判断の記録
- **RLS vs SECURITY DEFINER の使い分け**: 統計 RPC と運用 RPC は SECURITY DEFINER + `is_admin()` ガードで二重化。ユーザー詳細の多表参照（step_progress 等）は admin SELECT policies を付与してクライアント側から並列に fetch する方針とした。理由: 詳細は 11 テーブルに渡り、全部 RPC にすると型定義と保守コストが増える。policies で絞る方が読み出しの柔軟性が高い。
- **DAU の定義**: `point_history.created_at` を JST で日付丸めしたユニークユーザー数として定義。理由: ポイント獲得は「意味ある学習行動」の代表値で、step_progress の updated_at より網羅的（デイリー・コードドクター・ミニプロ・コードリーディングも拾える）。
- **auth.users.email 取得**: `authenticated` role からは auth.users を直接 SELECT できないため、`admin_list_users()` SECURITY DEFINER 内で `profiles join auth.users` して email を返す。ユーザー詳細画面では email を表示しない（プライバシーに配慮）。
- **exactOptionalPropertyTypes 対策**: `SummaryCard` の optional `accent` に `undefined` を渡すと TS2375 で落ちるため、呼び出し側で条件付きスプレッド `{...(cond ? { accent: '...' } : {})}` とした（M2 で feedback filter に使った同パターン）。

### 発見・ハマり・気づき
- admin_list_users の返却型に `email text` を入れると auth.users との結合で RLS 回避ができるが、SECURITY DEFINER 関数内で閉じ込めれば問題にならない（search_path を public に固定することで schema 攻撃も防げる）。
- DAU の generate_series 結合は `left join` で対応しないと 0 人の日が欠落する。最初 `inner join` で書いて値が 0 の日が表示されず気づいた。
- ステップ別完了率は「着手ユーザー中の完了割合」と定義（step_progress に行がある = 着手）。「全登録ユーザー中の完了割合」にすると新規ステップほど率が下がり比較困難になる。
- test mock: `from('user_feedback')` のように単一テーブルに複数のチェーン（count-only と eq 付き）が同時に走るケースでは、`select()` が返すオブジェクトに `then` と `eq` の両方を持たせる thenable を作る必要あり（adminStatsService.test.ts の createUserFeedderBuilder）。

### エラー・障害
- 初回 typecheck: AdminStatsPage で `accent: string | undefined` が `exactOptionalPropertyTypes` 下で optional プロパティに代入不可。条件付きスプレッドで解消。
- 初回 lint: adminStatsService.test.ts で unused-vars（`callCount`, `_col`, `_val`）3 件。mock を簡素化（bare eq シグネチャ）して解消。
